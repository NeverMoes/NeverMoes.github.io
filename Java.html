常见编码方式

  1. ASCII 码
    总共有 128 个，用 1 个字节的低 7 位表示， 0 ~ 31 是控制字符如换行、回车、删除等，32 ~ 126 是打印字符，可以通过键盘输入并且能够显示出来。

  2. ISO-8859-1
    128 个字符显然是不够用的，所以 ISO 组织在 ASCII 的基础上扩展，他们是 ISO-8859-1 至 ISO-8859-15，前者涵盖大多数字符，应用最广。ISO-8859-1 仍是单字节编码，它总归能表示 256 个字符。

  3. GB2312
    它是双字节编码，总的编码范围是 A1 ~ F7，其中 A1 ~ A9 是符号区，总共包含 682 个符号；B0 ~ F7 是汉字区，包含 6763 个汉字。

  4. GBk
    GBK 为《汉字内码扩展规范》，为 GB2312 的扩展，它的编码范围是 8140 ~ FEFE（去掉XX7F），总共有 23940 个码位，能表示 21003 个汉字，和 GB2312的编码兼容，不会有乱码。

  5. UTF-16
    它具体定义了 Unicode 字符在计算机中的存取方法。UTF-16 用两个字节来表示 Unicode 的转化格式，它采用定长的表示方法，即不论什么字符用两个字节表示。两个字节是 16 个 bit，所以叫 UTF-16。它表示字符非常方便，没两个字节表示一个字符，这就大大简化了字符串操作。

  6. UTF-8
    虽说 UTF-16 统一采用两个字节表示一个字符很简单方便，但是很大一部分字符用一个字节就可以表示，如果用两个字节表示，存储空间放大了一倍，在网络带宽有限的情况下会增加网络传输的流量。UTF-8 采用了一种变长技术，每个编码区域有不同的字码长度不同类型的字符可以由 1 ~ 6 个字节组成。

Java 中的编解码

首先要理解如果我们在源码中写入一个字符串常量，那么它会以什么样的方式最终被载入 JVM 内存内。

  1. .java (任意编码)，源码可以是任意方式的编码
  2. .javac (Unicode)，字节码一定会被编码成 Unicode
  3. JVM (Unicode)，内存中也是使用 Unicode

在我们编写代码过程中需要注意的就是源码到字节码过程中的编码转化。
需要注意几个点：

  1. 注意设置 IDE 编辑编码。
  2. getBytes() 会用一个系统相关的 Charset.defaultCharset() 来进行编码。
  3. 字节码编译过程中需要使用 javac -encoding 来指定编码，否则使用上面的默认编码。字符串常量也是在这里被指定编码方式的。
  4. maven 中的定义方式 <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>

即以内存为中心保证与外部交互时读取和写入内存的方式的都是 Unicode 编码即可。

编解码接口

内存方式

String s  = "字符串"；
byte[] b = s.getBytes("UTF-8");
String n = new String(b, "UTF-8");

Charset charset = Charset.forName("UTF-8");
ByteBuffer byteBuffer = charset.encode(string);
CharBuffer charBuffer = charset.decode(byteBuffer)

文件方式

String file = "c:/stream.txt"; 
String charset = "UTF-8"; 
// 写字符换转成字节流
FileOutputStream outputStream = new FileOutputStream(file); 
OutputStreamWriter writer = new OutputStreamWriter( 
    outputStream, charset); 
try { 
    writer.write("这是要保存的中文字符"); 
} finally { 
    writer.close(); 
} 
// 读取字节转换成字符
FileInputStream inputStream = new FileInputStream(file); 
InputStreamReader reader = new InputStreamReader( 
    inputStream, charset); 
StringBuffer buffer = new StringBuffer(); 
char[] buf = new char[64]; 
int count = 0; 
try { 
    while ((count = reader.read(buf)) != -1) { 
        buffer.append(buffer, 0, count); 
} 
} finally { 
    reader.close(); 
}

JavaWeb 中的编解码

URL

编码：
对于编码的情况，可能根据浏览器不同也会有不同的情况，详细可以查看：
关于URL编码 - 阮一峰

这里以我本机的 chrome 浏览器为例。这个 URL 会被编码成
http://localhost:8080/examples/servlets/servlet/%E5%90%9B%E5%B1%B1?author=%E5%90%9B%E5%B1%B1
注意有些浏览器的 Query String 和 Path Info 的编码可能是不一致。
URL 的编码规范 RFC3986 规定将非 ASCII 字符按照某种编码格式编码成 16 进制数字然后将每个 16 进制表示的字节前加上“%”。

解码：
回到后端看看这个编码是如何被解码的。
首先以 Tomcat 为例：
Path Info 的解码过程：

  1. org.apache.coyote.HTTP11.InternalInputBuffer::parseRequestLine 将接受到的 Byte[] 数组放进 Request 中，但是并没有进行解码。
  2. org.apache.catalina.connector.CoyoteAdapter::convertURI 方法中对其进行了解码。
    1. 解码字符集定义在 <Connector URIEncoding=”UTF-8”/> 中
    2. 如果没有定义，默认按ISO-8859-1，进行处理。

Query String 的解码过程：

  1. request.getParameter 第一次被调用时进行解码。
  2. org.apache.catalina.connector.Request::parseParameters 解码的方法。
  3. 解码字符集定义在 <Connector useBodyEncodingForURI=”true”/> ，如果此项为 true 则会使用 Header 中 ContentType 中定义的 Charset。否则为ISO-8859-1

HTTP Header

解码：

  1. request.getHeader 第一次被调用时进行解码。
  2. MessageBytes::toString 解码方法
  3. 默认字符集为 ISO-8859-1，且不能更改。
  4. 如果一定要使用非 ASCII 字符，使用 URLEncoder 编码进 Header。

POST 表单

编码：
浏览器会使用 ContentType 的 Charset 进行编码。

解码：

  1. request.getParameter 第一次被调用时进行解码。
  2. 字符集为 ContentType 的 Charset。
  3. 也可以通过 request.setCharacterEncoding(charset) 来设置。

HTTP Body

注意这里的编码是由服务端来完成的，解码是浏览器。
编码：

  1. response.setCharacterEncoding 设置字符集，会覆盖 request.getCharacterEncoding 的值。
  2. 并且通过 Header 的 Content-Type 返回给浏览器进行解码。

解码：

  1. Content-Type 的 charset 来解码。
  2. 如果不存在使用 <meta HTTP-equiv="Content-Type" content="text/html; charset=GBK" /> 中的 charset 。
  3. 如果也没用定义使用默认编码。
