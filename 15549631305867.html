Java 并发 - final 重排

  • 参考

final 的使用场景

final 关键字可以修饰变量，方法，类。final 在多线程场景下会存在重排问题。这里以final 修饰对象的不同来讨论一下这个问题。

final 修饰基本类型和引用类型的区别在于基本类型不可变，引用指向不可变，但是引用指向的对象状态可以改变。

  • 类变量有两个地方可以赋值，且只能在这两个地方赋值：
    • 在静态初始块
    • 在声明类变量时
  • 实例变量有三个地方可以赋值，且只能在这三个地方赋值
    • 非静态初始块
    • 声明实例变量时
    • 构造器
  • 局部变量可以不再声明时赋值，但是只有一次赋值机会。
  • 宏变量，在同时满足以下三个条件时，final 变量就会变成一个宏变量。
    1. 使用 final 修饰
    2. 定义变量时就指定初值
    3. 初值在编译时可以唯一指定
      比如：String s = "NeverMoes"。
      注意：当程序中使用到该宏变量的地方会直接替换成该变量的值。
  • 方法
    • final 修饰的方法不能被子类修改但是可以被子类重载。
  • 类
    • 修饰类时不能被集成

final 中的重排

基本类型重排

public class FinalDemo {
    private int a;  //普通域
    private final int b; //final域
    private static FinalDemo finalDemo;
    public FinalDemo() { //构造函数
        a = 1; // 1. 写普通域
        b = 2; // 2. 写final域
    }
    public static void writer() { 
        finalDemo = new FinalDemo();
    }
    public static void reader() {
        FinalDemo demo = finalDemo; // 3.读对象引用
        int a = demo.a;    //4.读普通域
        int b = demo.b;    //5.读final域
    }
}
/*
这里假设有两个线程一个执行 writer 方法，一个执行 reader 方法。
*/

write 方法实际上做了两件事（注意这两件事没有先后顺序）：

  • 构造了一个FinalDemo对象。
  • 把这个对象赋值给成员变量finalDemo。

这里给出了一种可能的执行顺序：

  • a 域被重排到了构造函数之外。
  • b 域在构造函数是内正确的赋值了。
  • b 域被正确读取。
  • a 度被读到的是一个未初始化的值。

这里就引出了 final 域的重排规则：
写 final 域的规则为：禁止 final 域被重排到构造函数之外。即编译器会在 final 域写之后，构造函数返回之前，插入一个 storestore 屏障。
这条写 final 域规则确保了：在对象引用被其他线程可见之前，对象的 final 域已经被正确初始化了。但是普通域不能被保证。
读 final 域的规则为：初次读对象引用和初次读该对象的 final 域，这两个操作禁止重排。
那么读 final 域规则确保了：在读一个对象的 final 域之前，一定会读包含这个 final 域的对象的引用

于是这两条规则共同确保了：基本类型的 final 域一定会被正常初始化写入后再被读取。

引用类型重排

当 final 域为基本类型时，重排的过程我们已经很清楚了，但是涉及到引用类型要注意了。
因为引用类型会比基本类型多一种操作。引用类型不止有引用本身的读写，还有引用对象域的读写。
所以对于final 修饰引用对象，会有一个额外的重排约束：final 修饰对象成员域的写入，与随后在构造函数之外把这个对象的引用赋给一个引用变量，这两个操作不能重排。
举个例子：

public class FinalReferenceDemo {
    final int[] arrays;
    private FinalReferenceDemo finalReferenceDemo;

    public FinalReferenceDemo() {
        arrays = new int[1];  //1
        arrays[0] = 1;        //2
    }

    public void writerOne() {
        finalReferenceDemo = new FinalReferenceDemo(); //3
    }

    public void writerTwo() {
        arrays[0] = 2;  //4
    }

    public void reader() {
        if (finalReferenceDemo != null) {  //5
            int temp = finalReferenceDemo.arrays[0];  //6
        }
    }
}
/*
线程以下顺序启动
假设 A 线程执行 writeOne
线程 B 执行 writeTwo
线程 C 执行 reader
*/

这里给出一种可能的执行情况：

即 final 域的重排规则可以保证 线程 C 可以看得到 线程 A 对数组元素的写入，但是不能保证线程 C 可以看到线程 B 的写入。如果想要保证，可以加 volatile 修饰。

小结一下就是：final 修饰的域可以保证在构造器中被正确初始化后再被读取。

final 逸出

虽然编译器保证了在其他地方获得对象引用之前， final 域已经被正确初始化了，但是要有前提条件，就是在构造函数之中不能让这个被构造的方法可见。即不能发生 this 逸出。

public class FinalReferenceEscapeDemo {
    private final int a;
    private FinalReferenceEscapeDemo referenceDemo;

    public FinalReferenceEscapeDemo() {
        a = 1;  //1
        referenceDemo = this; //2
    }

    public void writer() {
        new FinalReferenceEscapeDemo();
    }

    public void reader() {
        if (referenceDemo != null) {  //3
            int temp = referenceDemo.a; //4
        }
    }
}

如果发生了 this 逸出，那么编译器对 final 域的约束可能不能保证。
