适用场景

主要是高性能和高并发的场景。

  • KV 存储
  • 缓存
  • 消息中间件
  • 分布式锁

性能优化

redis 快主要是因为存在几个性能优化：

  • 内存数据库，不用频繁 io 操作。
  • 单线程，没有上下文切换消耗。
  • 高效的数据结构。
  • 合理的数据编码。

数据结构的分析在其他文章中。

存在问题

  1. 缓存和数据库双写一致性问题
  2. 缓存雪崩问题
  3. 缓存击穿问题
  4. 缓存的并发竞争问题

过期与淘汰策略

redis 对过期的 key 使用定期删除+惰性删除的机制。

  • 定期删除
    • redis 每隔 100ms 会对随机的 key 进行检查，有过期 key 则删除。
  • 惰性删除
    • 使用 key 时检查一下是否过期，过期则删除。

内存淘汰机制

  • 从以从已设置过期时间的数据集中挑选
    • volatile-lru：最近最少使用的数据淘汰
    • volatile-ttl：将要过期的数据淘汰
    • volatile-random：任意选择数据淘汰
  • allkeys-lru：（最常用）的当内存不足以容纳新写入数据时，在键空间中，移除最近最少使用的 key。
  • allkeys-random的当内存不足以容纳新写入数据时，在键空间中随机移除 key。
  • no-eviction：禁止驱逐数据，当内存不足以容纳新写入数据时，新写入操作会报错。

持久化机制：

  • RDB（快照），即保存某个时间点内存中副本。
  • AOF，主流方案，即追加修改数据的记录。

缓存相关

缓存雪崩

缓存同一时间大面积失效，造成大面积请求落到数据库上，导致数据库崩溃。

简单的解决方案就是：在缓存的时候给过期时间加上一个随机值，这样就会大幅度的减少缓存在同一时间过期。

  • 事发前：实现Redis的高可用(主从架构+Sentinel 或者Redis Cluster)，尽量避免Redis挂掉这种情况发生。
  • 事发中：万一Redis真的挂了，我们可以设置本地缓存(ehcache)+限流(hystrix)，尽量避免我们的数据库被干掉(起码能保证我们的服务还是能正常工作的)
  • 事发后：redis持久化，重启后自动从磁盘上加载数据，快速恢复缓存数据。

缓存穿透

黑客故意请求不在缓存中的数据，导致所有请求都落到数据库上。

  • 布隆过滤，将所有数据哈希存到一个足够大的 bitmap 中，一定不存在的数据也会过滤。
  • 缓存空数据，即就算一个数据的查询结果为空也缓存，但是缓存时间较短。

并发竞争 Key

如果在 redis 集群的情况下，尽量不要使用事务。
那么就存在两种情况：

  1. 对 key 的操作不要求顺序：
    • 准备一个分布式锁，得到锁的可以操作
  2. 如果要求顺序：
    • 写入数据时记录时间
    • 将 set 方法变成串行访问

事务

使用事务一般分为两步：

  • redis的事务可以分为两步，定义事务和执行事务。使用multi命令开启一个事务，然后把要执行的所有命令都依次排上去。这就定义好了一个事务。此时使用exec命令来执行这个事务，或使用discard命令来放弃这个事务。
  • 你可能希望在你的事务开始前，你关心的key不想被别人操作，那么可以使用watch命令来监视这些key，如果开始执行前这些key被其它命令操作了则会取消事务的。也可以使用unwatch命令来取消对这些key的监视。

Redis 的事务跟普通的数据库事务有所不同，具有以下的特点：

  1. 如果开始执行事务前出错，则所有命令都不执行
  2. 一旦开始，则保证所有命令一次性按顺序执行完而不被打断
  3. 如果执行过程中遇到错误，会继续执行下去，不会停止的
  4. 对于执行过程中遇到错误，是不会进行回滚的

所以 Redis 的事务是不能保证原子性的。

单线程模型

  • redis集群的每个节点里只有一个线程负责接受和执行所有客户端发送的请求。技术上使用多路复用I/O，使用Linux的epoll函数，这样一个线程就可以管理很多socket连接。
    除此之外，选择单线程还有以下这些原因：
  • redis都是对内存的操作，速度极快（10W+QPS）
  • 整体的时间主要都是消耗在了网络的传输上
  • 如果使用了多线程，则需要多线程同步，这样实现起来会变的复杂
  • 线程的加锁时间甚至都超过了对内存操作的时间
  • 多线程上下文频繁的切换需要消耗更多的CPU时间
  • 还有就是单线程天然支持原子操作，而且单线程的代码写起来更简单
