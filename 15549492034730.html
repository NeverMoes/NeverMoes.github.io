作用范围

根据 synchronized 使用方法的不同，锁住的对象也是不同的。

底层实现

Monitor 对象

与 synchronized 实现相关的一些信息会记录在对象头中的 monitor 对象中。
在 Java 中，对象包含三块：对象头、实例数据、填充数据，其中对象头中就包含 Mark Word，Mark Word 一般存储对象的 hashCode、GC分代年龄以及锁信息，锁信息就包含指向互斥量（重量级锁）的指针，指向了一个 monitor 对象，具体如下：

ObjectMonitor() {
    _header       = NULL;
    _count        = 0; //记录个数
    _waiters      = 0,
    _recursions   = 0;
    _object       = NULL;
    _owner        = NULL;
    _WaitSet      = NULL; //处于wait状态的线程，会被加入到_WaitSet
    _WaitSetLock  = 0 ;
    _Responsible  = NULL ;
    _succ         = NULL ;
    _cxq          = NULL ;
    FreeNext      = NULL ;
    _EntryList    = NULL ; //处于等待锁block状态的线程，会被加入到该列表
    _SpinFreq     = 0 ;
    _SpinClock    = 0 ;
    OwnerIsThread = 0 ;
}

核心组件

  1. Wait Set：哪些调用 wait 方法被阻塞的线程被放置在这里；
  2. Contention List：竞争队列，所有请求锁的线程首先被放在这个竞争队列中；
  3. Entry List：Contention List 中那些有资格成为候选资源的线程被移动到 Entry List 中；
  4. OnDeck：任意时刻，最多只有一个线程正在竞争锁资源，该线程被成为 OnDeck；
  5. Owner：当前已经获取到所资源的线程被称为 Owner；
  6. !Owner：当前释放锁的线程。

竞争过程

  1. JVM 每次从队列的尾部取出一个数据用于锁竞争候选者（ OnDeck ）， 但是并发情况下，ContentionList 会被大量的并发线程进行 CAS 访问，为了降低对尾部元素的竞争，JVM 会将一部分线程移动到 EntryList 中作为候选竞争线程。
  2. Owner 线程会在 unlock 时，将 ContentionList 中的部分线程迁移到 EntryList 中，并指定 EntryList 中的某个线程为 OnDeck 线程（一般是最先进去的那个线程）。
  3. Owner 线程并不直接把锁 传递给 OnDeck 线程， 而是把锁竞争的权利交给 OnDeck ，OnDeck 需要重新竞争锁。这样虽然牺牲了一些公平性，但是能极大的提升系统的吞吐量，在JVM 中，也把这种选择行为称之为“竞争切换”
  4. OnDeck 线程获取到锁资源后会变为 Owner 线程，而没有得到锁资源的仍然停留在 EntryList中。如果 Owner 线程被 wait 方法阻塞，则转移到 WaitSet 队列中，直到某个时刻通过 notify 或者 notifyAll 唤醒，会重新进去 EntryList 中。
  5. 处于 ContentionList、EntryList、WaitSet 中的线程都处于阻塞状态，该阻塞是由操作系统来完成的（Linux 内核下采用 pthread_mutex_lock 内核函数实现的）。

从这个流程可以看出几点：

  1. Synchronized 是非公平锁。 Synchronized 在线程进入 ContentionList 时，等待的线程会先尝试自旋获取锁，如果获取不到就进入 ContentionList，这明显对于已经进入队列的线程是不公平的，还有一个不公平的事情就是自旋获取锁的线程还可能直接抢占 OnDeck 线程的锁资源。
  2. 每个对象都有个 monitor 对象，加锁就是在竞争 monitor 对象。

锁优化

Jdk 1.6 之后对于 synchronized 进行了大量的优化。引入了偏向锁，轻量级锁，以及膨胀机制。

锁的记录

锁的信息记录在 Java 对象头中，不同的锁状态，对象头保存的信息是不同的。

锁膨胀过程

那么锁一共会有四种状态：无锁，偏向锁，轻量级锁，重量级锁。
锁可以升级但是不可以降级。

无锁

无锁不会对资源进行锁定，所有线程都可以访问共享资源，但是只有一个线程可以成功。这里就用了一个无锁或者说乐观锁算法：CAS算法。它假定线程对共享资源的访问不会发生冲突，或者发生冲突就一直重试到不发生冲突为止。简单来理解就是一个乐观锁，不会阻塞，在冲突的时候会发生自旋。

偏向锁

大多数情况下，锁不会存在竞争而是由同一线程多次获得，为了让线程降低获得锁的代价，而引入了偏向锁。

偏向锁的获取：当线程访问同步块并获得锁时，会在对象头和栈帧中记录线程ID。以后该线程进入相同同步块时只要检查一下记录的线程ID是否相同就可以获得锁了。
偏向锁的释放：偏向锁的释放需要等待一个全局安全点（即一个没有字节码在运行的时间点），JVM 会暂停拥有偏向锁的线程，并撤销偏向锁将其恢复到无锁或者轻量级锁的状态。

轻量级锁

当偏向锁被其他线程访问时偏向锁会升级为轻量级锁。
其他线程会通过自旋的方式来获取锁，并不会阻塞。

重量级锁

当线程等待轻量级锁自旋次数过多或有第三个线程访问锁轻量级锁会升级为重量级锁。重量级锁就是操作系统提供的 Mutex 锁。
